5. All Pages — Routes, Purpose, Components Used
Page 1: Doctor Terminal — /
Purpose: Doctor selects diagnostic tests for a patient and generates an invoice + payment link.
Components mounted:
•	widgets/app-shell/DesktopSidebar (desktop)
•	widgets/app-shell/MobileBottomNav (mobile)
•	widgets/page-header/PageHeader ("Doctor's Order Terminal")
•	features/order-management/ui/QuickAddGrid
•	features/order-management/ui/TestSearchBar
•	features/order-management/ui/TestCatalogTabs → renders TestCatalogList → renders entities/test/ui/TestListItem
•	features/order-management/ui/OrderSummary (right panel / bottom sheet on mobile)
API calls:
1.	GET /api/tests?category=hematology → populate catalog
2.	POST /api/orders → on "Generate Instant Payment & QR"
3.	POST /api/invoices → get payment_link
State (Zustand useOrderStore):
{ selectedTests: DiagnosticTest[], patientName: string, 
  orderId: string | null, invoiceUrl: string | null }

Page 2: Lab Dashboard — /dashboard
Purpose: Financial ledger. Today's revenue, settled splits, active orders. Transaction table with split engine expansion.
Components mounted:
•	widgets/app-shell/DesktopSidebar
•	widgets/app-shell/MobileBottomNav
•	widgets/page-header/PageHeader ("Lab Dashboard")
•	features/lab-dashboard/ui/RevenueStatCards
•	features/lab-dashboard/ui/TransactionTable → renders TransactionTableRow → expands to SplitEngineExpander
API calls:
1.	GET /api/dashboard?provider_id=1 → revenue stats + transaction list
State (Zustand useDashboardStore):
{ totalRevenue: number, settledSplits: number, activeOrders: number,
  transactions: Transaction[], expandedRow: string | null }

Page 3: Tech Fulfillment — /tech
Purpose: Lab technician sees queue of paid, verified orders. Can Start Test, Mark Sample, or Complete Order.
Components mounted:
•	widgets/app-shell/DesktopSidebar
•	widgets/app-shell/MobileBottomNav
•	widgets/page-header/PageHeader ("Technician Fulfillment")
•	features/tech-fulfillment/ui/ActiveDepartmentBanner
•	features/tech-fulfillment/ui/DepartmentFilterTabs
•	features/tech-fulfillment/ui/OrderCardMobile (mobile) OR OrdersTableDesktop (desktop) — switched via useMediaQuery
API calls:
1.	GET /api/orders?status=paid → active queue (NOTE: Ayo needs to confirm this endpoint or use dashboard data)
2.	PATCH /api/orders/:id/status → update order status (stretch — may mock for demo)
State (Zustand useFulfillmentStore):
{ activeOrders: Order[], activeDepartment: string, 
  activeFilter: 'queue' | 'history', searchQuery: string }

Page 4: Patient Invoice — /pay/[invoiceId]
Purpose: Patient-facing payment page. Shows invoice summary and triggers Interswitch Web Checkout.
Components mounted:
•	features/payment/ui/InvoiceHeader
•	features/payment/ui/InvoicePatientInfo
•	features/payment/ui/InvoiceLineItems
•	features/payment/ui/InvoiceTotals
•	features/payment/ui/PayButton → opens Interswitch Web Checkout widget
API calls:
1.	GET /api/invoices/:invoiceId → fetch invoice details (patient name, test list, total)
2.	After Interswitch callback: GET /api/payment/verify?transaction_reference=TXN-... → confirm payment
3.	On success → redirect to /pay/[invoiceId]/success
State (Zustand usePaymentStore):
{ invoice: Invoice | null, paymentStatus: 'idle' | 'loading' | 'success' | 'failed' }
Interswitch Integration Note: The Interswitch Web Checkout is loaded via their JS SDK script tag in layout.tsx. The PayButton calls window.webpayCheckout({ ... }) with the txnref, amount, merchantCode from the env vars, then handles the callback.
<!-- Add to app/layout.tsx <head> -->
<script src="https://newwebpay.interswitchng.com/plugin/browse-pay.js"></script>

Page 5: Payment Success — /pay/[invoiceId]/success
Purpose: Confirmation screen after Interswitch verifies payment.
Components mounted:
•	features/payment/ui/PaymentSuccessBanner
•	features/payment/ui/PaymentReceiptDetails
•	features/payment/ui/ResultsReadyNotice
•	features/payment/ui/DownloadReceiptButton
Data source: Query params from Interswitch callback (txnref, amount, resp) + one verification API call.

shared/hooks/useMediaQuery.ts
export function useIsMobile() {
  const [isMobile, setIsMobile] = useState(false);
  useEffect(() => {
    const mq = window.matchMedia('(max-width: 768px)');
    setIsMobile(mq.matches);
    mq.addEventListener('change', (e) => setIsMobile(e.matches));
  }, []);
  return isMobile;
}

14. Interswitch Web Checkout Integration
The PayButton component must trigger the Interswitch Web Checkout modal:
// features/payment/ui/PayButton.tsx
declare global {
  interface Window { webpayCheckout: (config: object) => void; }
}

const handlePay = () => {
  window.webpayCheckout({
    merchant_code: process.env.NEXT_PUBLIC_INTERSWITCH_MERCHANT_CODE,
    pay_item_id: '101',           // from Interswitch dashboard
    txn_ref: invoice.id,          // the TXN-... reference
    amount: invoice.amount * 100, // in kobo
    currency: 566,                // NGN
    site_redirect_url: `${process.env.NEXT_PUBLIC_FRONTEND_URL}/pay/${invoice.id}/success`,
    mode: 'TEST',                 // switch to 'LIVE' for production
    onComplete: (response) => {
      if (response.resp === '00') {
        router.push(`/pay/${invoice.id}/success?txnref=${invoice.id}`);
      }
    },
    onClose: () => toast('Payment cancelled'),
  });
};
